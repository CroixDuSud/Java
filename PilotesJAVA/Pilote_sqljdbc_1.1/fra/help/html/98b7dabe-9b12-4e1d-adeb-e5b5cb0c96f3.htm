L'accès simultané désigne les diverses techniques utilisées pour préserver l'intégrité de la base de données lorsque plusieurs utilisateurs mettent à jour les lignes en même temps. Un accès simultané incorrect peut générer des problèmes tels que des lectures de mauvaise qualité, des lectures fantômes et des lectures non renouvelées. Le pilote JDBC de Microsoft SQL Server 2005 intègre des interfaces avec toutes les autres techniques utilisées par SQL Server pour résoudre ces problèmes.

Remarque : Pour plus d'informations sur la simultanéité de SQL Server, voir « Gestion d'accès simultanés aux données » dans la documentation en ligne de SQL Server.

Le pilote JDBC prend en charge les types d'accès simultanés suivants :

Type d'accès simultané Caractéristiques Verrous de ligne Description

CONCUR_READ_ONLY

Lecture seule

Non

Les mises à jour à l'aide du curseur ne sont pas autorisées et aucun verrou n'est appliqué aux lignes qui composent le jeu de résultats.

CONCUR_UPDATABLE

Lecture écriture optimistes

Non

La base de données opère sur la base de l'hypothèse que des conflits de lignes sont improbables, mais possibles. L'intégrité des lignes est contrôlée à l'aide d'une comparaison d'horodatage.

CONCUR_SS_SCROLL_LOCKS

Lecture écriture pessimistes

Oui

La base de données opère sur la base de l'hypothèse que des conflits de lignes sont probables. L'intégrité des lignes est assurée par un verrouillage de ligne.

CONCUR_SS_OPTIMISTIC_CC

Lecture écriture optimistes

Non

La base de données opère sur la base de l'hypothèse que des conflits de lignes sont improbables, mais possibles. L'intégrité des lignes est contrôlée à l'aide d'une comparaison d'horodatage.

Pour SQL Server 2005, le serveur définit ce paramètre comme CONCUR_SS_OPTIMISTIC_CCVAL si la table ne contient pas de colonne timestamp.

Pour SQL Server 2000, si la table sous-jacente contient une colonne timestamp, le paramètre OPTIMISTIC WITH ROW VERSIONING est utilisé si le paramètre OPTIMISTIC WITH VALUES est spécifié. Si le paramètre OPTIMISTIC WITH ROW VERSIONING est spécifié et si la table ne contient pas d'horodatages, le paramètre OPTIMISTIC WITH VALUES est utilisé.

CONCUR_SS_OPTIMISTIC_CCVAL

Lecture écriture optimistes

Non

La base de données opère sur la base de l'hypothèse que des conflits de lignes sont improbables, mais possibles. L'intégrité des lignes est contrôlée à l'aide d'une comparaison de données de lignes.

Les jeux de résultats ne sont pas actualisables

Un jeu de résultats est actualisable lorsque des lignes peuvent y être insérées, mises à jour et supprimées. Dans les cas suivants, SQL Server ne peut pas créer de curseur actualisable. L'exception générée est « Le jeu de résultats ne peut pas être mis à jour ».

Cause Description Remède

Aucune instruction n'est créée à l'aide de la syntaxe JDBC 2.0

JDBC 2.0 utilise de nouvelles méthodes pour la création d'instructions. En cas d'utilisation de la syntaxe JDBC 1.0, le jeu de résultats est en lecture seule par défaut.

Spécifiez le type de jeu de résultats et la simultanéité lors de la création de l'instruction.

L'instruction est créée à l'aide du paramètre TYPE_SCROLL_INSENSITIVE.

SQL Server crée un curseur de capture instantanée statique. Celui-ci est déconnecté des lignes de la table sous-jacente afin de protéger le curseur contre les mises à jour de lignes par d'autres utilisateurs.

Utilisez TYPE_SCROLL_SENSITIVE, TYPE_SS_SCROLL_KEYSET, TYPE_SS_SCROLL_DYNAMIC ou TYPE_FORWARD_ONLY avec CONCUR_UPDATABLE afin d'éviter de créer un curseur statique.

La conception de la table exclut l'utilisation d'un curseur KEYSET ou DYNAMIC

La table sous-jacente ne contient pas de clé unique pour permettre à SQL Server d'identifier une ligne sans équivoque.

Ajoutez des clés uniques à la table afin d'obtenir une identification unique de chaque ligne.
