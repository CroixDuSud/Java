Lorsque vous créez une application qui accède à des données, vous devez partir de l'hypothèse que toutes les entrées utilisateur sont malveillantes jusqu'à preuve du contraire. À défaut de ce faire, vous rendriez votre application vulnérable aux attaques. Un type d'attaque susceptible de se produire est une injection SQL, où un code nuisible est ajouté à des chaînes qui sont ultérieurement transmises à une instance de SQL Server en vue de leur analyse et de leur exécution. Pour éviter ce type d'attaque, vous devez utiliser des procédures stockées avec des paramètres si c'est possible et toujours valider les entrées utilisateur.

La validation d'entrées utilisateur dans le code client est importante afin de ne pas perdre de temps en boucles de connexion avec le serveur. Il est également important de valider les paramètres des procédures stockées sur le serveur afin de capter les entrées qui ne sont pas valides et qui ignorent la validation coté client.

Pour plus d'informations sur l'injection SQL et la manière de l'éviter, voir la rubrique « Injection SQL » de la documentation en ligne de SQL Server 2005. Pour plus d'informations sur la validation des paramètres de procédure stockée, voir la rubrique « Procédures stockées (Moteur de base de données) » et les rubriques connexes de la documentation en ligne de SQL Server 2005.
